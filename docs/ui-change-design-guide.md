# UI 修改设计指南

本文用于约束 `milkTCompendium` 的界面调整。它借鉴 `systematic-design-prompt.json.pdf` 的分层设计方法，但改写为适合当前 SwiftUI / iOS 17+ 项目的规则。

## 可借鉴的部分

PDF 里最值得采用的不是具体视觉风格，而是系统化思路：

- 从设计哲学开始，再落到 token、组件、布局、响应式、文案、实现和 QA。
- 每个视觉元素都要有功能理由，装饰不能先于信息表达。
- 用排版、位置、字号、权重建立层级，少靠额外装饰制造重点。
- 色彩按功能使用：默认克制，只有状态、操作、风险、分歧等信息需要强调时才使用强调色。
- spacing、圆角、阴影、动效应形成可复用规则，而不是每个页面临时发挥。
- 质量检查要变成问题清单：能不能删掉、是否对齐、是否遮挡、是否表达真实数据、是否能在小屏成立。

不适合直接照搬的部分：

- Web 产品页的 hero、滚动叙事、光标系统、hover 规则不适合 iOS 主流程。
- Swiss 12 栏网页网格不能直接替代 SwiftUI 的安全区、手势区域和 Dynamic Type。
- 过度强调“产品展示”的插画逻辑，不适合当前以记录、贴纸、评分天梯和共享对比为核心的工具型 App。

## Level 0: 产品设计哲学

### 核心定位

`milkTCompendium` 是一个“喝过什么、好不好喝、和谁的口味有什么不同”的奶茶图鉴。

界面优先服务三件事：

- 快速记录一杯饮品。
- 在评分天梯里形成可浏览、可回忆的贴纸图鉴。
- 和朋友交换图鉴后，看见共同饮品的共识与分歧。

### 设计原则

- 图鉴感优先：饮品贴纸是主角，界面容器要让贴纸和评分关系清楚出现。
- 信息诚实：评分、杯数、品牌、甜度、冰度、限定状态都必须按真实语义展示，不用无意义图形强行美化。
- 少而明确：按钮、标签、浮层只保留当前任务需要的内容；不要用说明文字弥补结构不清。
- 操作可逆可确认：删除、导入、覆盖、丢弃草稿等高风险操作必须明确提示。
- 中文优先：所有面向用户的 UI 文案使用中文，短句、口语化、少解释。

## Level 1: Design Tokens

### 色彩

优先使用系统色，保持浅色模式下的清爽图鉴质感：

- 页面背景：`Color(.systemGroupedBackground)`。
- 主要卡片：`.white` 或 `.white.opacity(0.94...0.97)`。
- 次级信息面：`Color(.secondarySystemGroupedBackground)`。
- 主操作：黑色或系统 prominent button，避免大量彩色按钮并列。
- 弱强调：`.secondary`、`.tertiary`、黑色 4%-8% 透明度。
- 状态色：红色只用于删除/风险；绿色用于成功/低分歧；橙色用于处理中、提醒或中等分歧。

不要把棕色、橙色、蓝色等变成大面积主题色。奶茶感可以来自贴纸和内容，不靠整页奶茶色背景。

### 字体层级

使用 SwiftUI 系统字体，优先保留 Dynamic Type 兼容性。

- 页面级标题：`.headline`、`.title3`，仅在确实需要建立页面身份时使用。
- 卡片主信息：`.subheadline.weight(.semibold)` 或 18-22pt system rounded。
- 评分数字：使用 `.monospacedDigit()`，保证小数对齐。
- 标签和技术性数据：`.caption`、`.caption2`，可用 `.weight(.semibold/.bold)` 增强可扫读性。
- 不要用过大的标题抢走贴纸、评分轴和对比图谱的注意力。

### 间距

以 4pt / 8pt 节奏为基础：

- 页面边距：12-16pt。
- 卡片内部：10-14pt。
- 相关元素间距：6-10pt。
- 不同信息组：12-18pt。
- 浮层与屏幕边缘：至少 14pt，并避开底部操作区和安全区。

天梯、画布、贴纸碰撞框这类几何值不只属于视觉 token，修改时必须同步检查手势命中、缩放和碰撞规则。

### 圆角与阴影

- 小输入/标签：10-14pt。
- 表单卡片和面板：16-20pt。
- 浮层详情卡：22-24pt。
- 阴影只用于浮层、拖拽态、预览放大等层级变化；普通卡片优先用背景和 1pt 描边建立边界。

## Level 2: Component Patterns

### 评分天梯

评分天梯是首页主视觉，不是背景装饰。

- 贴纸位置必须首先表达评分位置。
- 缩放、拖动、点击、长按删除要和视觉位置一致。
- 标签只在放大后辅助识别，不能为了显示标签破坏贴纸排布。
- 修改图标尺寸、列距、zIndex、标签 reveal、碰撞框时，必须参考 `docs/ladder-layout.md`。

### 顶部控制区

顶部控制区承载图鉴切换、搜索、筛选、交换、导入导出等高频入口。

- 它应该像工具坞，不像导航栏标题区。
- 当前图鉴名称和数量状态要可扫读。
- 搜索和筛选展开时不能遮挡天梯核心手势区域太久。
- 临时测试入口必须显式标记，不要混入正式功能层级。

### 记录表单

记录表单服务“快速完成一杯记录”：

- 照片/贴纸预览和评分放在最上方。
- 品牌、名称、甜度、冰度、限定、杯数、备注按填写频率组织。
- 处理照片时必须有明确 loading 态，不能让用户误以为卡住。
- 重复饮品提示要给出清晰选择：加一杯、修正并加一杯、忽略、取消。

### 浮层详情卡

浮层用于短暂查看，不应变成第二个详情页。

- 背景遮罩要弱，只表达“当前聚焦”。
- 卡片内优先展示品牌、名称、贴纸、评分、甜度、冰度、杯数、备注。
- 可编辑入口只在本地图鉴饮品出现。
- 点按空白关闭，预览贴纸用 sheet。

### 共享对比页

共享对比页表达“共同饮品感受图谱”。

- 第一眼应看到整体趋势。
- 同一饮品内每个人的评分差异必须可见，不能只放进详情。
- 支持 2 人、3 人、5 人时结构都要成立。
- 饮品变多后要更像整体图谱，不应退化成拥挤列表。
- 修改前先读 `docs/shared-comparison-design-notes.md`。

## Level 3: Layout Rules

### 屏幕结构

- 首页：评分天梯全屏铺底，顶部工具坞和右下记录入口浮在其上。
- 记录页：垂直滚动表单，信息组以卡片分段，首屏必须看到照片和评分。
- 交换页：状态、附近设备、导入导出入口分块展示，避免用户分不清“交换”和“备份”。
- 对比页：图谱主体优先，header 只提供当前上下文和轻量操作。

### 对齐

- 表单文本左对齐，数字使用等宽数字。
- 卡片内标签和值必须有稳定对齐轴。
- 多列对比信息左右距离保持视觉对称。
- 图谱或天梯中的空间方向必须有真实数据含义，不能只是为了排得好看。

### 小屏适配

- 不靠省略号硬撑关键内容；必要时换行、降低密度或拆成 sheet。
- 交互目标尽量不小于 44pt。
- 顶部浮层、底部按钮和系统安全区不能互相遮挡。
- 键盘出现时，表单字段和保存按钮仍应可到达。

## Level 4: Interaction Rules

- 动效只用于状态变化、层级变化和手势反馈。
- 首页贴纸点击打开详情，长按/拖拽进入删除流程，删除必须确认。
- 缩放画布的状态不能跨图鉴污染；不要用延迟回调掩盖 viewport 生命周期问题。
- Sheet 用于短流程或聚焦任务；NavigationDestination 用于有明确页面层级的编辑和对比。
- 高风险操作使用 confirmation dialog 或 alert，文案说清后果。

## Level 5: Content Rules

### 文案语气

- 中文、短句、直接。
- 按用户动作命名按钮，例如“记录”“修改”“丢弃”“继续编辑”。
- 空状态给下一步，不写长说明。
- 错误文案说明当前发生了什么，以及用户能做什么。

### 信息优先级

- Must know：饮品名称、品牌、评分、贴纸、当前图鉴/对比对象。
- Should know：甜度、冰度、杯数、限定、备注、饮用时间。
- Nice to know：导入导出细节、交换统计、算法解释、维护性提示。

## Level 6: Implementation Guidelines

- 不引入新依赖，除非用户确认。
- 修改前先读相关 Swift 文件和 docs 中对应约束。
- 优先使用 SwiftUI 原生组件和状态管理；保持现有 `@StateObject`、`@ObservedObject`、`@Query` 等模式一致。
- 对独立 UI 片段抽小 view，但不要为了抽象而抽象。
- 自绘、`UIViewRepresentable`、坐标转换、缓存和手势状态属于高风险区，改动后必须实际验证。
- 用户可见文案保持中文。
- 不要用兜底 UI 掩盖主流程错误；先找清楚根因。

## Level 7: QA Checklist

每次 UI 修改至少检查：

- 这个元素是否有功能理由？能删吗？
- 信息层级是否一眼清楚？
- 文字、贴纸、评分点、按钮是否重叠？
- 小屏和大屏是否都可用？
- 交互目标是否容易点中？
- 动效是否服务状态变化，而不是装饰？
- 删除、导入、覆盖、丢弃等风险操作是否有确认？
- 中文文案是否自然、短、准确？
- 是否沿用现有颜色、间距、圆角、字体模式？
- 是否通过 Swift 类型检查或 Xcode build？

天梯相关改动追加检查：

- 常见密集评分段能自然排布。
- 放大后标签不会大面积消失或遮挡贴纸。
- 点击、长按拖拽、删除确认、缩放、切换图鉴都正常。
- 切换共享图鉴和对比对象后，画布不会继承错误 offset、inset 或 zoom state。

对比页相关改动追加检查：

- 5-8 个共同饮品时，单杯差异清楚。
- 30 个共同饮品时，整体趋势仍成立。
- 2 人、3 人、5 人时，多人结构自然扩展。
- 不读说明也能理解共识和分歧。

## UI 修改 Prompt 模板

需要让 Codex 或其他协作者改 UI 时，可以按这个格式描述：

```text
目标：修改哪个页面/组件，用户要完成什么任务。
当前问题：具体哪里不清楚、不好点、不好看、遮挡或表达错误。
约束：必须保留哪些行为、数据语义、手势、文案或文件边界。
风格：沿用本指南的颜色、间距、圆角、字体和动效规则。
验证：说明需要跑哪些场景，例如小屏、空状态、密集评分、多人对比。
禁止：不引入新依赖；不改无关文件；不使用兜底掩盖主流程错误。
```

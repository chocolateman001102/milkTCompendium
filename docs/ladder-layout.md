# 天梯首页布局注意事项

`milkTCompendium/CollectionView.swift` 里的首页天梯同时负责：

- 图标排布：饮品贴纸不能互相重叠。
- 文字显示：放大后显示饮品名称和品牌。
- 手势命中：点击、长按拖拽删除仍要对准图标/文字区域。

这三个模块高度耦合。每次修改首页天梯、图标尺寸、缩放比例、碰撞框、列距、zIndex、或标签显示逻辑时，都必须联动检查本文件记录的约束。

## 缩放视野与跨图鉴切换约束

首页天梯和图鉴对比天梯都通过 `UIViewRepresentable` 包装 `UIScrollView`。它们共享 `ZoomCanvasViewportController`，因此以下规则必须同时适用于两个页面。

### 曾出现的故障与根因

在交换或导入他人图鉴后，切换“我的 / 对方的”图鉴可能出现以下组合问题：双指缩放后视野向左上偏移、可拖动范围被错误限制，且即使未进行新的缩放操作，切换回来后仍会继承异常状态。

根因不是饮品数据本身，而是两项原生视图状态问题叠加：

1. SwiftUI 会复用同一个 `UIScrollView` 来展示另一份画布；上一图鉴的 `contentOffset`、`contentInset` 和手势状态会在新画布的首个布局周期内残留。
2. 画布替换和缩放过程中，`UIScrollView.contentSize` 会短暂反映旧尺寸或未缩放尺寸。若用它推导居中 inset / 可拖动边界，下一次 pinch 会从错误坐标系开始。

### 当前实现规则

- `ZoomCanvasViewportController` 的 inset、边界和重置焦点必须只根据“画布原始尺寸 × 当前 zoomScale”计算，不能把 `scrollView.contentSize` 作为几何真值。
- 重置缩放时直接设置目标 `zoomScale`，再从内容坐标系计算并夹紧 `contentOffset`；不要使用依赖瞬时 `contentSize` 的 `zoom(to:)`。
- 不能通过多轮 `asyncAfter` 延迟重置来等待布局稳定。延迟回调可能在用户已经切到另一份图鉴后才执行，从而污染新页面。
- 每份天梯画布必须以 `contentSignature` 作为 SwiftUI identity，切换图鉴或对比对象时重建原生滚动视图，隔离上一份画布的 offset、inset、缩放和手势状态。
- `contentSignature` 不能退化成单纯的图鉴 ID。冷启动、共享图鉴异步读盘、附近交换覆盖同一 ownerID、过滤条件变化，都可能在“同一个图鉴 ID”下生成不同画布；这些情况下也必须重建或重新 reset viewport。
- 当 `contentSignature`、画布尺寸或显式 reset token 变化时，`updateUIView` 必须请求一次中心 reset，不能只对旧 `contentOffset` 做 clamp。clamp 只适用于普通缩放/拖动结束后的合法化。
- 缩放、拖动或减速结束后都要重新夹紧 content offset；当缩放后的画布小于视口时，对应轴必须保持居中。
- 首页长按拖拽饮品会临时关闭 `UIScrollView.isScrollEnabled`。任何手势取消、画布身份变化、尺寸变化、重置、切换图鉴或 SwiftUI 更新中断，都必须通过统一收尾路径恢复滚动，避免画布偶发无法拖动。

### 缩放回归场景

- 重启 App 后：我的图鉴缩放、拖动、缩回最小倍率均正常。
- 冷启动后等待共享图鉴异步加载完成，首次进入共享图鉴：视野应直接居中，不能依赖“切到我的再切回来”恢复。
- 通过附近交换覆盖已有同一 ownerID 的共享图鉴，并自动切到该图鉴：首次显示、缩放和拖动范围均应正常。
- 不进行缩放直接切至对方图鉴，再切回我的图鉴：视野不应向左上偏移，且拖动范围正常。
- 在我的图鉴和任一共享图鉴之间反复切换；每个页面分别缩放、拖动，再切换回来。
- 对比天梯中切换不同对比对象后，重复上述缩放和拖动检查。
- 在内容较少、画布在最小缩放下小于视口的情况下，画布应居中且不能被拖到左上角；放大后应获得完整、对称的可拖动范围。
- 首页长按饮品后分别测试：正常松手、拖到删除区弹确认、拖动过程中切换图鉴、打开/关闭交换页。之后画布仍应可以正常拖动和缩放。

## 本次布局调整

- 首页天梯的初始视觉位置通过 `CollectionView.initialLadderVisualYOffset` 下移绘制坐标，而不是依赖 `UIScrollView` 的初始 `contentOffset`。这样打开首页时会露出更多天梯顶部区域。
- `LadderMetrics` 接收 `verticalVisualOffset`，并同时下移 `plotTop` / `plotBottom`，保证评分轴和饮品节点整体同步移动。
- 画布高度必须为 `verticalVisualOffset` 预留同等空间，且 `plotBottom` 必须落在画布内；否则绘制内容和可拖动边界会不一致。
- `LadderLayoutProfile.stable.columnSpacing` 调整为 `82`，让多列饮品排布比原先更紧凑一些，同时保留碰撞检测的安全距离。

## 当前布局原则

1. 图标排布优先保证评分位置准确。
   - y 坐标由评分线性映射得到，不应为了避让做大幅垂直偏移。
   - `.95` 和 `.75` 这类相差 `0.20` 的邻近档位，应在无同档拥挤时可以同列容纳。

2. 图标碰撞和文字碰撞分阶段处理。
   - 第一阶段只用图标碰撞框决定图标位置。
   - 标签框不应参与图标排布，否则 `.95` 的文字保护区会把 `.75` 图标挤到其它列。
   - 第二阶段在图标位置固定后，再决定每个标签是否显示。

3. 文字模块的保护距离必须比图标模块更宽松。
   - 标签属于放大后的辅助信息，不能因为邻近图标太多而大面积全部隐藏。
   - 当前标签显示只和已显示标签互斥，不再避让所有其它图标；否则邻近评分段会导致几乎所有文字都不显示。
   - 如果未来重新加入 label-vs-icon 检查，必须只检查非常近的真实遮挡，不要使用宽大的全局保护框。

4. zIndex 与标签可见性相关。
   - 普通节点使用基于 y 位置的稳定层级，让上方节点盖住下方节点。
   - 拖拽节点保持最高层级。
   - 修改绘制顺序时要检查 2.95、2.75 等密集档位的文字是否被贴纸盖住。

## 修改首页天梯时的回归清单

每次动 `CollectionView.swift` 的首页天梯相关代码后，至少检查：

- `3.00 / 2.95 / 2.75 / 2.50` 能否自然排布。
- `.95` 与 `.75` 档位不会互相强制排斥到其它列。
- 放大到标签 reveal 区间后，邻近图标下面的文字不会大面积消失。
- 多个同评分饮品仍会左右/多列展开。
- 点击饮品卡片、长按拖拽删除仍正常。
- Xcode iOS Simulator build 通过。

## 相关代码入口

- `ladderCanvasSize(for:)`：画布尺寸、列数估算、同列评分间距约束。
- `ladderEntries(in:)`：两阶段布局，先排图标，再决定标签显示。
- `bestPlacement(...)`：图标候选位置选择。
- `isIconSafe(...)` / `isLabelSafe(...)`：碰撞规则。
- `LadderLayoutProfile.stable`：贴纸、标签、padding、列距的核心参数。
- `LadderDrinkNode` / `LadderStickerBadge`：真实视觉尺寸。
